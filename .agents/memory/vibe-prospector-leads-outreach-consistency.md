---
name: Vibe Prospector leads/outreach data consistency
description: What "in outreach" means, why status enums are duplicated in a few places, and where UI/UX parity between Leads and Outreach pages must be kept.
---

- The canonical predicate for "a lead is in outreach" is `outreach_mode not in [null, "none"] OR outreach_status != "not_started"`. It's used in three independent places that must stay aligned: `artifacts/dashboard/src/pages/outreach.tsx` (what's shown/counted), `artifacts/api-server/src/routes/leads.ts` `/leads/export` sales_caller filter (what a sales caller can export), and nowhere else — the dashboard home page intentionally does NOT show an "in outreach" count (it only shows pipeline_status buckets from `/leads/stats`). If a future feature needs an outreach count elsewhere, reuse this exact predicate rather than approximating it.
- `pipeline_status` (prefixed, ordered strings like `30_Ready_for_Outreach`) and `outreach_status` (plain slugs like `meeting_scheduled`) are two independent dimensions on the same Lead record, not duplicates of each other — don't try to unify their enums.
- Status/mode enum values (`PIPELINE_STATUSES`, `OUTREACH_STATUSES`, `OUTREACH_MODES` in `artifacts/dashboard/src/lib/constants.ts`) are duplicated as plain string-array allowlists in `artifacts/api-server/src/routes/leads.ts` (`VALID_STATUSES`, `VALID_OUTREACH_MODES`, `VALID_OUTREACH_STATUSES`) for request validation, and as bare `type: string` in `lib/api-spec/openapi.yaml` (no enum constraint). All three currently agree. If you add/rename a status value, update all three or they will silently drift — the openapi enum isn't a source of truth here since it doesn't constrain the string at all.
- The Outreach Tracker page (`artifacts/dashboard/src/pages/outreach.tsx`) was rebuilt to mirror the Leads page (`leads.tsx`) table structure/styling exactly (same Table components, column look, font-mono/uppercase header style, row-click-to-detail, inline status Select with color classes) instead of a card-list — keep new outreach features consistent with that table pattern rather than reverting to cards.
- `phone` and `mobile_speed_score` are properly typed (not `| null` cast to `any`) in the generated `Lead` type — don't reintroduce `(lead as any).phone` casts in dashboard pages, use `lead.phone` directly.
